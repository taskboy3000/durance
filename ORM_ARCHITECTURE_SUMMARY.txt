┌────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│ 1. ORM::DB - DATABASE CONNECTION MANAGER                                                                          │
├────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤

PRIMARY RESPONSIBILITY:
  • Manage database connections and pooling across the application

PUBLIC METHODS:
  • dbh() ........................... Get/create database handle with pooling
  • disconnect_all() ................ Disconnect all pooled handles  
  • isDSNValid() .................... Validate DSN without connecting

RESPONSIBILITIES COUNT: 2 (Connection pooling, Connection validation)

SRP COMPLIANCE: ✓ COMPLIANT (GOOD)
  → Only 1-2 related responsibilities
  → Focused on single concern: connection management
  → No business logic or query building
  → Clean separation of concerns

FRAMEWORK REQUIREMENTS MET:
  ✓ Lightweight .................... Minimal dependencies (only DBI)
  ✓ Convention over Configuration .. Derives from model package names
  ✓ Database Abstraction ........... Via DSN pattern
  ✓ ORM Operations Support ......... Provides dbh to models
  ✗ Relationship Management ........ Not applicable
  ✗ Schema Introspection/Migration . Not applicable
  ✗ Query Building ................. Not applicable
  ✗ Verbose SQL Logging ............ Not implemented
  ✗ Dry-run Mode ................... Not applicable

OBSERVATIONS:
  • Global state via %HANDLES package variable (per-class pooling)
  • Connection validation optional (good API design)
  • No driver-specific handling needed at this level


┌────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│ 2. ORM::DSL - DOMAIN SPECIFIC LANGUAGE FOR MODEL DEFINITION                                                       │
├────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤

PRIMARY RESPONSIBILITY:
  • Export functions for defining model structure (columns, relationships, validations)

PUBLIC FUNCTIONS EXPORTED:
  • column() ........................ Define model column with metadata
  • tablename() .................... Define table name
  • has_many() ..................... Define has-many relationship
  • belongs_to() ................... Define belongs-to relationship
  • validates() .................... Define validation rules

RESPONSIBILITIES COUNT: 5 (Column definition, Accessor generation, Relationship def, Relationship accessor gen, Validation)

SRP COMPLIANCE: ✗ VIOLATION (5 responsibilities - should be 1-2)
  ✗ Column metadata registration + accessor code generation (2 concerns)
  ✗ Relationship definition + method generation (2 concerns)
  ✗ Validation rule registration (1 concern)
  → These should be separated into distinct modules

FRAMEWORK REQUIREMENTS MET:
  ✓ Lightweight .................... No external dependencies
  ✓ Convention over Configuration .. Default foreign keys, naming conventions
  ✓ Relationship Management ........ has_many, belongs_to with auto foreign keys
  ✓ ORM Operations ................. Enables model definition
  ✗ Database Abstraction ........... Not applicable
  ✗ Schema Introspection/Migration . Not applicable
  ✗ Query Building ................. Not applicable
  ✗ Verbose SQL Logging ............ Not applicable
  ✗ Dry-run Mode ................... Not applicable

CRITICAL ISSUES:
  • Tight coupling to ORM::Model package variables (%COLUMN_META, %_validations)
  • Inline validation in generated setters (length, format, required, bool coercion)
  • Accessor generation via symbolic references (*{"${pkg}::${name}"})
  • Foreign key defaults hardcoded: "${parent_name}_id"
  • Mixing concerns: DSL parsing + code generation + validation


┌────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│ 3. ORM::Model - BASE CLASS FOR ORM MODELS (GOD OBJECT - MAJOR REFACTORING NEEDED)                                 │
├────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤

PRIMARY RESPONSIBILITY:
  • Provide CRUD operations and query interface for model instances

PUBLIC METHODS (22 TOTAL):
  Metadata Access:     columns(), column_meta(), primary_key(), validations(), attributes(), table()
  Relationships:       has_many_relations(), belongs_to_relations(), related_to(), all_relations()
  Lookup:              find(), all(), where(), count(), first()
  Mutation:            create(), insert(), update(), delete(), save()
  Utility:             to_hash(), db(), schema_name(), _db_class_for()

RESPONSIBILITIES COUNT: 8 (Too many!)
  1. CRUD operations (create, insert, update, delete, save)
  2. Query interface (where, find, all, count, first)
  3. Attribute/metadata access (columns, column_meta, primary_key, etc.)
  4. Relationship access (has_many_relations, belongs_to_relations, etc.)
  5. Validation access (validations method)
  6. DB instance management (db method)
  7. DB class derivation (_db_class_for)
  8. Schema name extraction (schema_name)

SRP COMPLIANCE: ✗ MAJOR VIOLATION (8 responsibilities - should be 1-2)
  ✗ This is a "God Object" with too many responsibilities
  → Should split: CRUD layer, Query layer, Metadata layer, Relationship layer

FRAMEWORK REQUIREMENTS MET:
  ✓ Convention over Configuration .. Auto db class derivation
  ✓ Database Abstraction ........... Queries through dbh
  ✓ Relationship Management ........ Relationship queries
  ✓ Query Building ................. where(), limit(), order() chaining
  ✓ ORM Operations (CRUD) ......... Full CRUD support
  ✗ Schema Introspection/Migration . Not applicable (Schema module handles)
  ✗ Lightweight .................... Complex dependency chain
  ✗ Verbose SQL Logging ............ No query timing captured

CRITICAL ISSUES:
  • God Object with 22 public methods (should be ~5-8)
  • Queries directly constructed in methods (no unified query builder)
  • Hard-coded timestamps (created_at, updated_at)
  • Metadata stored globally in package variables (fragile)
  • DB class derivation logic duplicated in Schema module
  • Lazy-loading of ResultSet class
  • Mix of class and instance methods
  • No query timing or execution tracking
  • BUILD method modifies args object


┌────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│ 4. ORM::ResultSet - CHAINABLE QUERY INTERFACE                                                                     │
├────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤

PRIMARY RESPONSIBILITY:
  • Build and execute queries with chainable methods

PUBLIC METHODS:
  • where() ........................ Add WHERE conditions (builder)
  • order() ........................ Add ORDER BY (builder)
  • limit() ........................ Add LIMIT (builder)
  • offset() ....................... Add OFFSET (builder)
  • add_joins() .................... Add JOIN clauses (builder)
  • all() .......................... Execute query, return results
  • first() ........................ Execute with LIMIT 1
  • count() ........................ Execute COUNT query

QUERY ATTRIBUTES:
  • class (ro, required) ........... Model class
  • conditions (rw) ............... WHERE conditions
  • order_by (rw) ................. ORDER BY clauses
  • limit_val (rw) ................ LIMIT value
  • offset_val (rw) ............... OFFSET value
  • join_specs (rw) ............... JOIN specifications

RESPONSIBILITIES COUNT: 3
  1. Query state management (conditions, order, limit, offset, joins)
  2. SQL generation (building SQL strings)
  3. Query execution (executing prepared statements)

SRP COMPLIANCE: ✗ VIOLATION (3 responsibilities - should be 1 per class)
  ✗ State management should be separate from SQL generation
  ✗ SQL generation should be separate from execution
  → Suggested split: QueryBuilder (state) → SQLGenerator → QueryExecutor

FRAMEWORK REQUIREMENTS MET:
  ✓ Lightweight .................... No external dependencies
  ✓ Convention over Configuration .. Auto joins from relationships
  ✓ Relationship Management ........ JOIN support via add_joins()
  ✓ Query Building ................. Chainable interface
  ✓ ORM Operations ................. Executes queries
  ✗ Database Abstraction ........... Uses low-level SQL concatenation
  ✗ Schema Introspection/Migration . Not applicable
  ✗ Verbose SQL Logging ............ No query timing captured
  ✗ Dry-run Mode ................... Not applicable

CRITICAL ISSUES:
  • SQL concatenation directly in code (manual string building)
  • JOIN logic complex with metadata lookup (75+ line all() method)
  • Duplicate WHERE clause building in all() and count()
  • No query logging or verbose mode
  • No execution time tracking
  • Relationship metadata validated for string names only
  • Helpful error messages (lists available relationships)


┌────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│ 5. ORM::Schema - SCHEMA MANAGEMENT AND MIGRATION                                                                  │
├────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤

PRIMARY RESPONSIBILITY:
  • Manage database schema creation and migration based on model definitions

PUBLIC METHODS (11 TOTAL):
  DDL Generation:     ddl_for_class()
  Table Operations:   create_table(), table_exists(), column_info(), table_info()
  Migration:          migrate(), migrate_all(), sync_table(), pending_changes()
  Validation:         schema_valid(), ensure_schema_valid()
  Model Discovery:    get_all_models_for_app()

PRIVATE METHODS (9):
  _detect_driver(), _type_for(), _column_sql(), _pk_sql(), _build_columns_def()
  _build_create_table_sql(), _get_dbh_for(), _db_class_for(), _get_app_library_basedir()

RESPONSIBILITIES COUNT: 7 (Too many!)
  1. DDL generation (CREATE TABLE SQL building)
  2. Type mapping (ORM type → SQL type conversion)
  3. Table introspection (column_info, table_info)
  4. Column introspection (analyzing existing tables)
  5. Migration logic (create vs alter decision)
  6. Model discovery (filesystem walk for models)
  7. Schema validation (pending_changes, schema_valid)

SRP COMPLIANCE: ✗ VIOLATION (7 responsibilities - should be 1-2)
  → Suggested split: DDLBuilder, TypeMapper, Introspector, ModelDiscovery, MigrationEngine

FRAMEWORK REQUIREMENTS MET:
  ✓ Lightweight .................... Uses File::Find, no external deps
  ✓ Convention over Configuration .. Derives DB class from model name
  ✓ Schema Introspection/Migration . Core feature, well implemented
  ✓ Verbose SQL Logging ............ Has optional logger callback (◐ partial)
  ✗ Database Abstraction ........... Works at SQL level
  ✗ Relationship Management ........ Not applicable
  ✗ Query Building ................. Not applicable
  ✗ ORM Operations ................. Not applicable
  ✗ Dry-run Mode ................... NOT IMPLEMENTED (REQUIREMENT NOT MET!)

CRITICAL ISSUES:
  • Type mapping hardcoded (only sqlite/mysql - not extensible)
  • Model discovery uses global @gModelClasses variable (fragile)
  • String escaping for DEFAULT values is fragile (checks for \D)
  • VARCHAR length hardcoded to 255 for MySQL
  • NO DRY-RUN MODE (Framework requirement from AGENTS.md not met!)
  • Logging is optional but no query timing
  • chdir + cwd restoration pattern (works but fragile)
  • Duplicate db_class_for logic with Model module
  • Auto-increment syntax varies per driver (handled well)

     • Log raw SQL before execution
     • Capture execution time

     • Add to ORM::Schema for migration preview
     • Allow inspection of DDL without execution
  
  3. Extract SQL Builder
     • Create ORM::Query::Builder to centralize SQL generation
     • Remove duplicate WHERE building (all() vs count())
     • Consolidate schema SQL generation

     • Create ORM::Model::Metadata for metadata access
     • Keep CRUD in ORM::Model
     • Move query methods to use ResultSet

     • Relationship definition vs method generation

     • Consolidate _db_class_for (Model, Schema)
     • Extract type mapping to ORM::Type
  
  3. Registry Improvement
     • Consider hashref-based registry vs package variables
     • Better isolation for testing
